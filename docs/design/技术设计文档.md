# 技术设计文档

> **项目名称：** 基于智慧家居场景的大语言模型智能体科研应用一体机  
> **合同依据：** 江门定开产品合同（40 项技术要求）  
> **更新日期：** 2026-04-07

---

## 一、产品定位

一台本地离线运行的 AI 智能体服务器，部署在客户现场，服务于智慧家居场景的科研与演示。核心特征：

- **全本地化**：模型推理、智能体运行、知识库检索均在本地完成，不依赖外网
- **全开源合规**：Ollama(MIT) + Dify CE(Apache 2.0) + Home Assistant(Apache 2.0)
- **支持二次开发**：FastAPI + Next.js 全栈脚手架，提供 Code Server 和 JupyterLab

### 5.2 真实硬件对接预留

当前系统已完全预留真实硬件对接能力，代码改动极小：

```
┌─────────────┐      ┌──────────────────┐      ┌────────────────────┐
│  前端/AI     │ ──→  │  FastAPI 后端     │ ──→  │  Home Assistant    │
│  (不变)      │      │  (不变)           │      │  (已对接)          │
└─────────────┘      └──────────────────┘      └────────┬───────────┘
                                                         │
                                              ┌──────────▼───────────┐
                                              │ 真实设备（ESP32 等）   │
                                              │ 通过 HA 集成接入       │
                                              └──────────────────────┘
```

**对接步骤**：
1. 在 HA 中安装设备集成（如 ESPHome / Zigbee2MQTT / MQTT）
2. 设备自动注册为 HA 实体（无需改后端代码）
3. 在后端 `DEVICE_REGISTRY` 中添加真实 entity_id 映射
4. 前端设备控制页自动显示新设备

## 六、技术调研备忘

### 6.1 Dify 关键设计决策

- **模型供应商**：Dify 配置 Ollama 时地址用 Docker 内网 `http://ollama:11434`
- **RAG 流程**：文档 → Dify 分段 → bge-m3 向量化 → Weaviate 存储 → 语义检索
- **Workflow 编排**：主 Chatflow 通过 HTTP 节点调用子 Workflow

### 6.2 HA 模板实体格式

HA 2026 中废弃了 legacy 的 `brightness_template`，必须使用 modern format：

```yaml
template:
  - light:
      - name: "客厅灯"
        unique_id: sandbox_living_room
        state: "{{ states('input_boolean.light_living_room') }}"
        level: "{{ states('input_number.light_living_room_brightness') | int }}"
        set_level:
          - action: input_number.set_value
            target:
              entity_id: input_number.light_living_room_brightness
            data:
              value: "{{ brightness }}"
```
